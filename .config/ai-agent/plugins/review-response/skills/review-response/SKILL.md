以下の手順でgithubのPRに対するレビューを読んで、対応を行ってください。

# 手順

## 0. githubのオーナやリポジトリを確認する。
`git remote -v`

以降では、リポジトリ名等は以下のタグ表記を使用する。

<Repository>リポジトリ名</Repository>
<Owner>オーナー名</Owner>
<PR_Number>PRの番号</PR_Number>

## 1. 現在のブランチのPRの番号を確認する。
`gh pr status --json number,title,headRefName`

ブランチが見つからない場合は確認が必要です。

## 2. スクリプトで未解決のレビューを取得する。
```bash
~/.config/ai-agent/plugins/review-response/skills/review-response/scripts/list-unresolved-reviews.sh <Owner> <Repository> <PR_Number>
```

出力例：
```
---
File: path/to/file.sh:45
Thread ID: PRRT_kwDOHZmkFc5n_6u4
Status: Unresolved, Outdated=false
First Comment ID: 2660705204
Thread (1 comments):
  [reviewer] レビューコメント
```

## 3. 手順2の出力の各指摘について、対応と返答案を準備する

各レビュースレッドは「修正依頼」ではなく、**問題提起・根拠・原因推定・suggest change・期待する対応**が混ざった入力として扱う。自動レビューの指摘も同じで、指摘を機械的に修正したり、機械的に否定してはならない。

判断の中心は、長い現状説明ではなく、以下を分離して短く検証すること:

- レビュアーの問題提起は、どの条件で成立するか
- レビュアーの原因推定は正しいか
- suggest change は問題を正しいレイヤーで解くか
- このPRで対応する価値が、作業時間・リスク・スコープに見合うか
- 対応しない場合、問題をどう追跡または明確に却下するか

### 3-1. レビュー主張を分解する
対象コメントを読み、次を分けて抽出する。コメント本文の言い換えではなく、対象コード・呼び出し元・設定・依存関係を読んだ上で書く。

- **問題提起**: レビュアーが危険視している状態、壊れる振る舞い、不足している性質は何か
- **根拠**: レビュアーが示しているコード上の事実、実行結果、仕様、経験則は何か
- **原因推定**: レビュアーが「なぜ起きる」と考えているか
- **suggest change**: 具体的な変更提案がある場合、その変更は何を置き換えるか
- **期待する対応**: バグ修正、設計変更、テスト追加、説明、追跡Issue化、いずれを求めているか

問題提起、原因推定、suggest change、期待する対応を同一視してはならない。問題提起が正しくても原因推定やsuggest changeが誤っている場合がある。

### 3-2. 正しさと再現性を判定する
指摘の正しさは二値で扱わない。最低限、以下の分類を明示する。

- **再現性**: 常時再現 / 条件付きで再現 / コード上は成立するが未再現 / 可能性はあるが未確認 / 前提不一致
- **成立条件**: バージョン依存、時間依存、外部サービス依存、OS・シェル・ブラウザ依存、権限依存、データ量依存、並行実行依存、設定依存のどれか
- **原因推定**: 正しい / 部分的に正しい / 誤り / 未確定
- **確認根拠**: 実行で再現した、コード上の制御フローで成立する、仕様・ドキュメントで確認した、または未確認であること

再現できないだけで「不正確」と判断してはならない。低頻度、バージョン差、時刻、外部状態、権限、データ条件でのみ成立する指摘は、再現性が低いのではなく成立条件が狭い可能性がある。成立条件が特定できない場合は、結論ではなく追加調査または暫定対応として扱う。

### 3-3. 影響範囲とシステムの脆弱性を評価する
レビュー指摘が事実として成立しても、採るべき対応は影響範囲とシステム状態で変わる。

- **影響対象**: 顧客、利用者、運用者、開発者、CI、将来の保守者のどれに影響するか
- **影響の種類**: セキュリティ、データ破壊、停止、誤表示、性能劣化、開発体験、保守性、将来変更の阻害のどれか
- **頻度と重大度**: 低頻度でも重大度が高いか、高頻度でも影響が限定的か
- **既存の脆弱性**: そのバグが単発の欠陥か、脆弱な設計・境界・テスト不足の症状か

システムが脆弱な場合、レビューで見つかった個別バグだけを小さく直しても意味が薄いことがある。その場合は、局所パッチではなく根本対応、最小緩和と追跡、またはこのPRでの修正打ち切りを検討する。無限にレビュー指摘と小修正を繰り返すことは、作業時間とレビュー品質を浪費するため避ける。

### 3-4. suggest change と解決策を評価する
suggest change は、指摘の正しさとは独立に評価する。以下を確認する。

- 問題を根本的に解くか、症状だけを隠すか
- 正しい責務レイヤーで解いているか
- 呼び出し元、呼び出し先、型、設定、テスト、運用手順に副作用がないか
- 既存設計と整合するか、将来の変更容易性を下げないか
- 同じ根本原因の別症状を残さないか
- 自動レビューの局所最適な提案をそのまま採用していないか

「動く」「差分が小さい」「レビューを閉じられる」は採用理由として不十分。suggest change が不適切な場合は、問題自体をどう扱うべきかを必ず特定する。

### 3-5. 作業時間と費用対効果を見積もる
対応方針を決める前に、作業時間と検証コストを見積もる。細かい時間を装う必要はないが、根拠のない「軽微」「大きい」は禁止する。

- **小**: 対象が局所的で、仕様確認・修正・検証が同一PR内で完結する
- **中**: 複数ファイル、呼び出し関係、テスト追加、既存仕様の確認が必要
- **大**: 設計変更、移行、複数コンポーネント、広い回帰確認、関係者調整が必要
- **不明**: 再現条件や根本原因が未確定で、まず調査時間が必要

見積もりには、修正時間だけでなく、調査、検証、レビュー、リリース影響、将来の負債化、対応しない場合のリスクを含める。同じ根本原因から複数のレビュー指摘が出ている場合は、個別対応の合計時間ではなく、根本対応または打ち切りの判断を優先する。

### 3-6. 全体最適として対応方針を決める
以下のいずれかに分類し、理由を短く書く。

- **本PRで実装**: 問題が成立し、対応がPR目的と整合し、作業時間とリスクが見合う
- **本PRで最小緩和し、別途追跡**: 影響は無視できないが、根本対応はPRスコープや作業時間を超える
- **別PR・別Issueで追跡**: 問題は成立するが、このPRで触ると目的・責務・リスクが崩れる
- **別レイヤーへ申し送り**: 所有者や責務が明確に別で、このPR内では適切に解けない
- **返信のみ**: 指摘の前提が不一致、または自動レビュー由来の低価値な指摘で、対応価値がない
- **判断保留**: 成立条件、影響、作業時間の不確実性が大きく、追加調査やユーザ判断が必要

「スコープ外」「責務違い」は、何もしない理由ではない。問題が成立するなら、追跡先、申し送り先、または対応しない合理的理由を明記する。逆に、問題が成立しても、脆弱なシステムに対する局所パッチが全体最適でないなら、その場で修正しない判断もあり得る。

### 3-7. 対応の実施
方針が明確に決まったら、必要に応じて修正して commit, push し、返答案をドラフトする。本PRで実装しない場合も、追跡先、申し送り先、または対応しない理由を返答案に含める。

ユーザに選択肢を立てて投げるのは例外。ただし、次のような場合は選択肢を提示してよい。

- 根本対応が大きく、PRの目的やリリース時期に影響する
- 最小緩和と根本対応で、作業時間・安全性・将来負債のトレードオフが明確に割れる
- システム全体の脆弱性が原因で、個別レビュー対応を続けること自体の価値が低い
- 所有者、仕様、運用判断がコードだけでは決められない

この場合も「どうしましょう」と漠然と投げず、各案の作業量、効果、残るリスク、追跡方法を具体化して手順4で提示する。

## 4. 返答案・選択肢をまとめてユーザに確認する

手順3で用意した内容を、対応する `First Comment ID` / 対象ファイル位置と共に**まとめて一度だけ**ユーザに提示し、投稿可否・修正希望・選択（該当時）を確認する。ユーザからの承認を得るまでは `reply-review-comment.sh` を実行してはならない。

返答案がユーザとの会話で使用している言語と異なる場合は、ユーザ確認時は会話言語への訳を併記する（投稿時は原文のみ）。

提示フォーマット（方針が明確なケース）：
```
[連番] <File:Line>
コメント:
  [<reviewer>] <元のレビューコメント本文>
レビュー主張:
  <問題提起・原因推定・期待する対応を分けて短く記述>
成立条件・再現性:
  <常時/条件付き/未確認/前提不一致、バージョン・時間・環境・データ依存など>
影響:
  <対象者、影響種類、頻度と重大度、既存の脆弱性との関係>
suggest change の評価:
  <採用 / 修正して採用 / 不採用 / 提案なし> — <責務・副作用・根本解決性の根拠>
作業見積もり:
  <小 / 中 / 大 / 不明> — <調査・修正・検証・レビューに必要な作業の根拠>
判断:
  <本PRで実装 / 本PRで最小緩和し別途追跡 / 別PR・別Issueで追跡 / 別レイヤーへ申し送り / 返信のみ / 判断保留> — <全体最適としての根拠>
対応:
  <実施した変更の概要、または「本PRでは変更なし」+ 追跡方法>
返答案:
  <返答文>
```

提示フォーマット（選択を仰ぐケース）：
```
[連番] <File:Line>
コメント:
  [<reviewer>] <元のレビューコメント本文>
判断保留の理由:
  <成立条件・影響・作業時間・所有者など、コードだけで決められない事実>
選択肢:
  A. <案Aの内容（場所・方法・作業見積もり・効果・残るリスク・追跡方法）>
  B. <案Bの内容（場所・方法・作業見積もり・効果・残るリスク・追跡方法）>
  （他あれば追加）
判断材料:
  <選択を分けている事実（PR目的、リリース影響、システム脆弱性、累積負債、対応しない場合のリスクなど）>
```

## 5. 承認後に返答を投稿し、必要ならresolveする

### 返答
投稿する返答文には、対応するcommitハッシュを添えると親切（ユーザ確認用の提示には含めない）。

```bash
# 上記で確認したFirst Comment IDを使用
echo '[確認済みの返答文（必要に応じてcommitハッシュを付与）]' | ~/.config/ai-agent/plugins/review-response/skills/review-response/scripts/reply-review-comment.sh <Owner> <Repository> <PR_Number> ${COMMENT_ID}
```

### resolvedにマーク（必要な場合のみ）
```bash
# 上記で確認したThread IDを使用
~/.config/ai-agent/plugins/review-response/skills/review-response/scripts/resolve-review-thread.sh ${THREAD_ID}
```

## 注意事項

1. 取得制限
   - Review取得のスクリプトは未解決のレビュースレッドを最新20件まで取得
   - 各スレッドのコメントは最大10件まで取得

2. 認証
   - `gh`コマンドが自動的に認証情報を使用
   - プライベートリポジトリ等のアクセス権限が必要
