以下の手順でgithubのPRに対するレビューを読んで、対応を行ってください。

# 手順

## 0. githubのオーナやリポジトリを確認する。
`git remote -v`

以降では、リポジトリ名等は以下のタグ表記を使用する。

<Repository>リポジトリ名</Repository>
<Owner>オーナー名</Owner>
<PR_Number>PRの番号</PR_Number>

## 1. 現在のブランチのPRの番号を確認する。
`gh pr status --json number,title,headRefName`

ブランチが見つからない場合は確認が必要です。

## 2. スクリプトで未解決のレビューを取得する。
```bash
~/.config/ai-agent/plugins/review-response/skills/review-response/scripts/list-unresolved-reviews.sh <Owner> <Repository> <PR_Number>
```

出力例：
```
---
File: path/to/file.sh:45
Thread ID: PRRT_kwDOHZmkFc5n_6u4
Status: Unresolved, Outdated=false
First Comment ID: 2660705204
Thread (1 comments):
  [reviewer] レビューコメント
```

## 3. 手順2の出力の各指摘について、対応と返答案を準備する

各レビュースレッドは「修正依頼」ではなく、**問題提起・根拠・原因推定・suggest change・期待する対応**が混ざった入力として扱う。自動レビューの指摘も同じで、指摘を機械的に修正したり、機械的に否定してはならない。

この手順で確認する情報は、原則として内部判断に使う。ユーザに提示するのは、判断に必要な事実、推奨方針、確認が必要な論点だけに絞る。調査メモや思考過程をそのまま展開して、人間側の認知負荷を増やしてはならない。

### 3-1. 問題を理解する
対象コメントと対象コードを読み、レビューの問題提起が何かを特定する。コメント本文の言い換えではなく、対象コード・呼び出し元・設定・依存関係から確認する。

- **問題提起**: レビュアーが危険視している状態、壊れる振る舞い、不足している性質は何か
- **成立条件**: 常時成立するか、バージョン、時間、外部サービス、OS、権限、データ量、設定、並行実行などに依存するか
- **原因推定**: レビュアーの原因説明が正しいか、問題は正しいが原因は違うか
- **確認根拠**: 実行結果、コード上の制御フロー、仕様、既存方針のどれで確認したか

再現できないだけで「不正確」と判断してはならない。低頻度、バージョン差、時刻、外部状態、権限、データ条件でのみ成立する指摘は、再現性が低いのではなく成立条件が狭い可能性がある。成立条件が特定できない場合は、結論ではなく追加調査または暫定対応として扱う。

### 3-2. 解決策を理解する
suggest change がある場合は、問題提起とは独立に評価する。

- suggest change が何を解くか
- 問題を根本的に解くか、症状だけを隠すか
- 正しい責務レイヤーで解くか
- 既存設計や類似実装と整合するか
- 同じ根本原因の別症状を残さないか

問題提起が正しく、suggest change も局所的には動く場合でも、実装場所やPRスコープが適切とは限らない。特に外部API、認証、例外変換、永続化、権限、共通コンポーネントに関わる指摘では、責務境界を確認する。

### 3-3. 影響範囲と作業量を確認する
対応方針を決める前に、影響範囲と作業量を確認する。

- **影響範囲**: このPRの追加・変更箇所だけか、既存機能、共通基盤、他コンポーネント、将来の実装パターンに波及するか
- **影響対象**: 顧客、利用者、運用者、開発者、CI、将来の保守者のどれか
- **重大度**: セキュリティ、データ破壊、停止、情報漏洩、誤表示、性能劣化、保守性低下のどれに当たるか
- **作業量**: 小（局所修正）/ 中（複数ファイル・テスト・既存仕様確認）/ 大（設計変更・移行・広い回帰確認）/ 不明（成立条件や原因の追加調査が必要）

システムが脆弱な場合、個別バグだけを小さく直しても意味が薄いことがある。同じ根本原因から複数のレビュー指摘が出ている場合は、個別対応を続けるより、根本対応、最小緩和と追跡、またはこのPRでの修正打ち切りを検討する。

### 3-4. 対応方針を決める
事実から決められる部分は自律的に判断する。プロジェクト思想、PRスコープ、将来影響、作業量の重み付けに依存する部分は、推奨案を出した上でユーザに確認する。

方針は以下から選ぶ。

- **本PRで実装**: 問題が成立し、責務・スコープ・作業量がこのPRに収まる
- **本PRで最小緩和し、別途追跡**: 影響は無視できないが、根本対応はこのPRの範囲を超える
- **別PR・別Issueで追跡**: 問題は成立するが、このPRで触ると目的やリスクが崩れる
- **別レイヤーへ申し送り**: 所有者や責務が明確に別で、このPR内では適切に解けない
- **返信のみ**: 前提不一致、既存処理で対応済み、または対応価値がない
- **判断保留**: 方針がプロジェクト判断に依存し、ユーザ確認なしでは進めるべきでない

「スコープ外」「責務違い」は、問題を無視する理由ではない。問題が成立するなら、追跡先、申し送り先、または対応しない合理的理由を明記する。逆に、問題が成立しても、局所パッチが全体最適でないなら、その場で修正しない判断もあり得る。

### 3-5. 対応の実施
方針が明確に決まったら、必要に応じて修正して commit, push し、返答案をドラフトする。本PRで実装しない場合も、追跡先、申し送り先、または対応しない理由を返答案に含める。

次のような場合は、実装前にユーザへ確認する。

- 根本対応が大きく、PRの目的やリリース時期に影響する
- 最小緩和と根本対応で、作業時間・安全性・将来負債のトレードオフが明確に割れる
- システム全体の脆弱性が原因で、個別レビュー対応を続けること自体の価値が低い
- 所有者、仕様、運用判断がコードだけでは決められない

この場合も「どうしましょう」と漠然と投げず、推奨案、代替案、作業量、効果、残るリスク、追跡方法を短く具体化して手順4で提示する。

## 4. 返答案・選択肢をまとめてユーザに確認する

手順3で用意した内容を、対応する `First Comment ID` / 対象ファイル位置と共に**まとめて一度だけ**ユーザに提示し、投稿可否・修正希望・選択（該当時）を確認する。ユーザからの承認を得るまでは `reply-review-comment.sh` を実行してはならない。

返答案がユーザとの会話で使用している言語と異なる場合は、ユーザ確認時は会話言語への訳を併記する（投稿時は原文のみ）。

提示では、内部判断の全項目を出さない。1スレッドあたり、原則として以下の4項目に圧縮する。

提示フォーマット（方針が明確なケース）：
```
[連番] <File:Line>
コメント:
  [<reviewer>] <元のレビューコメント本文>
判断:
  <問題は成立 / 条件付きで成立 / 前提不一致 / 既存対応済み>。<根拠を1-2文>
影響:
  <判断に必要な場合のみ。責務、PRスコープ、作業量、将来影響を1-2文>
対応:
  <本PRで実装 / 最小緩和+追跡 / 別PR・別Issue / 別レイヤーへ申し送り / 返信のみ>。<理由を1文>
返答案:
  <返答文>
```

提示フォーマット（選択を仰ぐケース）：
```
[連番] <File:Line>
コメント:
  [<reviewer>] <元のレビューコメント本文>
判断:
  <問題は成立するが、実装場所・PRスコープ・作業量の重み付けがプロジェクト方針に依存することを短く記述>
推奨:
  <推奨案>。<作業量、効果、残るリスクを1-2文>
代替:
  <代替案が必要な場合のみ。何を優先すると代替案が妥当になるかを1文>
確認:
  <推奨案で進めてよいか、またはどの方針を優先するかを具体的に質問する>
```

## 5. 承認後に返答を投稿し、必要ならresolveする

### 返答
投稿する返答文には、対応するcommitハッシュを添えると親切（ユーザ確認用の提示には含めない）。

```bash
# 上記で確認したFirst Comment IDを使用
echo '[確認済みの返答文（必要に応じてcommitハッシュを付与）]' | ~/.config/ai-agent/plugins/review-response/skills/review-response/scripts/reply-review-comment.sh <Owner> <Repository> <PR_Number> ${COMMENT_ID}
```

### resolvedにマーク（必要な場合のみ）
```bash
# 上記で確認したThread IDを使用
~/.config/ai-agent/plugins/review-response/skills/review-response/scripts/resolve-review-thread.sh ${THREAD_ID}
```

## 注意事項

1. 取得制限
   - Review取得のスクリプトは未解決のレビュースレッドを最新20件まで取得
   - 各スレッドのコメントは最大10件まで取得

2. 認証
   - `gh`コマンドが自動的に認証情報を使用
   - プライベートリポジトリ等のアクセス権限が必要
