以下の手順でgithubのPRに対するレビューを読んで、対応を行ってください。

# 手順

## 0. githubのオーナやリポジトリを確認する。
`git remote -v`

以降では、リポジトリ名等は以下のタグ表記を使用する。

<Repository>リポジトリ名</Repository>
<Owner>オーナー名</Owner>
<PR_Number>PRの番号</PR_Number>

## 1. 現在のブランチのPRの番号を確認する。
`gh pr status --json number,title,headRefName`

ブランチが見つからない場合は確認が必要です。

## 2. スクリプトで未解決のレビューを取得する。
```bash
~/.config/ai-agent/plugins/review-response/skills/review-response/scripts/list-unresolved-reviews.sh <Owner> <Repository> <PR_Number>
```

出力例：
```
---
File: path/to/file.sh:45
Thread ID: PRRT_kwDOHZmkFc5n_6u4
Status: Unresolved, Outdated=false
First Comment ID: 2660705204
Thread (1 comments):
  [reviewer] レビューコメント
```

## 3. 手順2の出力の各指摘について、対応と返答案を準備する

レビュアーの指摘は正しいことが多いですが、指摘している問題が正しく、それに対するsuggest changeが解決策のhowとして間違っていなくても、それらが全体最適として正しいかどうかはまた別の話です。
機械的に修正や指摘を受け入れるべきではありません。またそれを踏まえて、あなたが判断したとしてもユーザやプロジェクトの意向と合致するかどうかもまた別の話です。

この手順で確認する情報は、原則として内部判断に使う。ユーザに提示するのは、判断に必要な事実、推奨方針、確認が必要な論点だけに絞る。
調査メモや思考過程をそのまま展開して、人間側の認知負荷を増やしてはならない。

### 3-1.  レビュアーの指摘を分析する
対象コメントと対象コードを読み、レビューの問題提起が何かを特定する。コメント本文の言い換えではなく、対象コード・呼び出し元・設定・依存関係から確認する。

- **問題提起**: レビュアーが危険視している状態、壊れる振る舞い、不足している性質は何か
- **成立条件**: 常時成立するか、バージョン、時間、外部サービス、権限、などの前提が間違っている可能性がないか
- **原因推定**: レビュアーの原因説明が正しいか、問題は正しいが原因は違うなど
- **確認根拠**: 実行結果、コード上の制御フロー、仕様、既存方針のどれで確認したか

再現できないだけで「不正確」と判断してはならない。
特定の条件でのみ成立する指摘は、再現性が低いのではなく成立条件が狭い可能性がある。成立条件が特定できない場合は、結論ではなく追加調査または暫定対応として扱う。

suggest change がある場合は、問題提起とは独立に評価する。

- 問題を根本的に解くか、特定の問題に限定した解決策か
- 正しい責務レイヤーで解くか
- 変更による他の影響範囲は考慮されているか
- 実装場所やPRスコープは正しいか

### 3-2. 影響範囲と作業量を確認する

- **問題の影響範囲**: この問題に対応しなかった場合にどれだけのリスクがあるか。誰にとっての問題か。
- **解決策の影響範囲**: このPRの追加・変更箇所だけか。他のコンポーネントに波及するか。将来的にこの実装の保守が負債になりうる可能性がないか。実装場所（レイヤー）は適切か。
- **作業量**: 小（局所修正）/ 中（複数ファイル・テスト・既存仕様確認）/ 大（設計変更・移行・広い回帰確認）/ 不明（成立条件や原因の追加調査が必要）

バグの指摘が正しくても、それを直すことが正しいとは限らない。根本原因の解決をした方が良いこともあれば、その指摘単体を治してもキリがないこともある。
全体としてあるべき姿を目指しつつ指摘の問題に対して全体最適を目指す必要がある。
ただし、プロジェクトによって優先順位が異なる場合があることには注意が必要。

以下のような場合は、対応方針の判断がプロジェクト全体に影響するため、3-3での確認時に特に留意する。

- 根本対応が大きく、PRの目的やリリース時期に影響する
- 最小緩和と根本対応で、作業時間・安全性・将来負債のトレードオフが明確に割れる
- システム全体の脆弱性が原因で、個別レビュー対応を続けること自体の価値が低い
- 所有者、仕様、運用判断がコードだけでは決められない

### 3-3. 対応方針をユーザに確認する
PRで対応するかissueに切り出すのか、どこまでを対応するのかなどを、プロジェクトの思想・進捗、スコープ、将来的な影響等を考慮したうえで、対応案をユーザに提示し合意を得る。
「どうしましょう」と漠然と投げず、推奨案、代替案、作業量、効果、残るリスク、追跡方法を短く具体化する。

「スコープ外」「責務違い」は、問題を解決しない理由にはならない。
問題が成立するなら、追跡先や対応しない合理的理由を明記する。
逆に、問題が成立しても、局所的パッチが全体最適でないなら、その場で修正しない判断もあり得る。

提示フォーマット
```
[連番] <File:Line>
<コメント>:
[<reviewer>] <元のレビューコメント本文>

<影響範囲・対応可否の判断>:
問題が成立しているか。問題自体・解決策による影響範囲。スコープや作業量的に対応可能かどうか
```

### 3-4. 対応の実施
ユーザと合意した方針に基づき、必要に応じて修正して commit, push し、返答案をドラフトする。本PRで実装しない場合も、追跡先、または対応しない理由を返答案に含める。

## 4. 返答案をユーザに確認する

3-4で作成した返答案を、対応する `First Comment ID` / 対象ファイル位置と共にユーザに提示し、投稿可否・修正希望を確認する。
ユーザからの承認を得るまでは `reply-review-comment.sh` を実行してはならない。

返答案がユーザとの会話で使用している言語と異なる場合は、ユーザ確認時は会話言語への訳を併記する（投稿時は原文のみ）。

提示フォーマット
```
[連番] <File:Line>
<コメント>:
[<reviewer>] <元のレビューコメント本文>

<返答案>
返答文
```

## 5. 承認後に返答を投稿し、必要ならresolveする

### 返答
投稿する返答文には、対応するcommitハッシュを添えると親切（ユーザ確認用の提示には含めない）。

```bash
# 上記で確認したFirst Comment IDを使用
echo '[確認済みの返答文（必要に応じてcommitハッシュを付与）]' | ~/.config/ai-agent/plugins/review-response/skills/review-response/scripts/reply-review-comment.sh <Owner> <Repository> <PR_Number> ${COMMENT_ID}
```

### resolvedにマーク（必要な場合のみ）
```bash
# 上記で確認したThread IDを使用
~/.config/ai-agent/plugins/review-response/skills/review-response/scripts/resolve-review-thread.sh ${THREAD_ID}
```

## 注意事項

1. 取得制限
   - Review取得のスクリプトは未解決のレビュースレッドを最新20件まで取得
   - 各スレッドのコメントは最大10件まで取得

2. 認証
   - `gh`コマンドが自動的に認証情報を使用
   - プライベートリポジトリ等のアクセス権限が必要
